什么时候重组：

1. State 发生变化时，只要 Composable 内部读取了某个 State，只要 state.value 变化，就会刷新读取了它的那部分代码
2. composable 传入的参数发生变化
3. 动画或副作用

State 和 collection 类似，是容器，其内部的值变化才会导致重组

derivedStateOf：防抖，内部数据变化，但是最终计算的结果不变，就不会通知下游

remember: 跨越重组的记忆

Compose 性能核心：减少不必要的重组和减少不必要的计算

compositionLocalOf 是动态的，当值发生变化时，只有那些真正读取了这个值的子组件才会重新组合，适用场景：值可能频繁变化

staticCompositionLocalOf （静态）：一旦这个值发生变化，整个 CompositionLocalProvider 内部的所有子组件都会被强制重新组合。速度快一点，没有订阅机制

CompositionLocalProvider 可以使内部的子树能像访问全局变量一样，对一些值的访问，避免了麻烦的参数传递



Synchronzied 换 Mutex

```
1.synchronzied 是阻塞的，Mutex 是挂起的，mutex 极大的提高了线程的利用率。
2. 防止死锁陷阱：
	在 synchronized 块里调用了一个 suspend 函数(如网络请求)
	后果：
		协程挂起，释放了线程
		但是 synchronized 还没退出，锁依然被持有
		其他协程想拿这个锁，就会被无限期阻塞
		很容易导致死锁问题
		Mutex 的好处，可以配合协程，放心的调用其他挂起函数
```

CompletableDeferred

```
Job (基类，只负责控制生命周期)
  ↑
Deferred<T> (接口，继承 Job，增加了 "返回值" 的概念)
  ↑
CompletableDeferred<T> (接口/实现，继承 Deferred，增加了 "手动填值" 的能力)

Job 只管任务是否完成、取消任务
Deferred 可以拿到结果
CompletableDeferred 可以手动的填入结果
async 发送请求 -> 造一个空的 Completable，保存起来 -> 收到消息 -> .complete(response) -> async协程收到结果并苏醒
```
