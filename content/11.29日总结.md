# 📅 入职第一周工作复盘：云游戏业务 Android 端起步

## 🚀 核心概况

**业务方向**：云游戏（Cloud Gaming）
**角色定位**：Android 独立开发（Owner）
**核心策略**：利用 AI 辅助降低试错成本，大胆引入前沿架构（Jetpack Compose + Clean Architecture），实现高效开发与技术探索的平衡。

## 🏗 技术架构演进

本周完成了项目基础架构的搭建，重点实践了现代化 Android 开发理念：

1. **UI 框架**：全面采用 **Jetpack Compose**，提升 UI 开发效率与声明式编程体验。
2. **构建系统**：使用 **Gradle Convention Plugins** 管理构建逻辑，统一模块配置，保持构建脚本整洁。
3. **架构模式**：向 **Clean Architecture** 演进，严格遵守依赖倒置原则（DIP）。引入 UseCase 层，解耦业务逻辑与 UI。利用 **Hilt** 实现依赖注入，优化模块间交互。
4. **核心技术栈**：从纯 CRUD 业务转向音视频领域，深入集成 **WebRTC**，探索云游戏场景下的低延迟流媒体传输。

## 🛠 问题排查与经验沉淀

### 1. Hilt 依赖注入的“隐形”陷阱

- **现象**：RtcModule 代码逻辑无误，但编译时报错，提示生成的 IndiePlayApp_HiltComponents 找不到 RtcModule 符号。
- **根因**：RtcModule 类缺少 package 声明（处于未命名包中）。Hilt 生成的 HiltWrapper_RtcModule 默认具有包私有访问权限，且同样位于未命名包。而在 Java 规范中，**有包名的类无法引用未命名包中的类**，导致 Hilt 组件无法链接该模块。
- **解决**：规范包结构，为所有类添加明确的 package 声明。

### 2. Gradle 构建效率优化

掌握了精准构建命令，避免全量编译，提升调试效率：

```bash
# 仅配置不执行任务，快速检查构建脚本错误
./gradlew :app:assembleDebug --dry-run
# 跳过测试任务，加速构建
./gradlew :core:webrtc:assembleDebug -x test
# 单独构建 WebRTC 核心模块
./gradlew :core:webrtc:assembleDebug
```
