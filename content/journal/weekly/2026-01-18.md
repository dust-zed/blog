### 📅 本周工作总结：架构减负与技术选型决断

**核心关键词：** 架构重构、FFI 优化、条件编译、技术止损

#### 1. 🚀 架构演进：模型层级“大一统”

痛点：

此前项目中存在严重的数据模型冗余，一条数据从底层到 UI 需要经历 4 层转换：

RustModel (底层) -> FfiModel (中间层) -> DomainModel (业务层) -> UiModel (前端)。

这导致每次新增字段都需要编写大量重复的 From/Into 样板代码，开发效率低下且维护成本高。

**尝试与突破：**

- **尝试一（失败）：** 最初试图利用 Rust 的 `pub use` 重导出机制复用结构体，但在 UniFFI 的宏展开和模块可见性规则下遇到编译阻碍。
- **尝试二（成功）：** 转向 **条件编译 (Conditional Compilation)** 方案。
  - 利用 `#[cfg_attr(feature = "ffi", derive(uniffi::Record))]`，让同一个 Struct 在不同编译目标下表现不同特性。
  - **成果：** 成功将 `RustModel` 与 `FfiModel` 合二为一。现在底层数据结构在开启 FFI 特性时，自动具备跨语言传输能力。
  - **收益：** 移除了中间层的转换逻辑，真正实现了“定义即接口”，极大降低了心智负担和代码量。

#### 2. 🛠 项目推进：前后端联调

- 完成了与后台接口的深度联调工作。
- 解决了 JSON 数据结构对齐中的若干细节问题，确保了多端数据传输的稳定性。
- （可选补充：虽然过程琐碎，但通过这次联调，进一步确认了利用 Rust 强类型系统在边界进行校验的必要性。）

#### 3. 🛑 技术决策：调整 WebRTC 路线

- **决策：** 在云游戏项目中正式放弃使用 `webrtc-rs` 库。
- **原因：** 经过评估，`webrtc-rs` 在现阶段的成熟度、Android 平台适配性以及云游戏所需的特定优化（如低延迟控制、硬件解码接入）方面，可能不如预期或维护成本过高。
- **后续计划：** 将探索更底层或更成熟的替代方案（如 C++ 原生库封装或 GStreamer 管道方案）来满足云游戏对高吞吐和低延迟的苛刻要求。

#### 4. 🔭 学习与展望：渲染技术栈

- **新目标：** 开始深入研究 **wgpu** (WebGPU for Rust) 和 **GPUI**。
- **动机：** 云游戏客户端不仅需要网络传输，还需要极高性能的渲染能力。`wgpu` 提供了跨平台的现代图形 API 抽象，而 `GPUI` 这种高性能 UI 框架的设计理念对构建下一代客户端有重要参考价值。

### 💡 Gemini 的复盘分析（供你参考）

1. **关于架构简化：** 你做的这个 `cfg_attr` 优化非常关键。在混合开发中，**“胶水层越薄越好”**。你把 4 层变成 3 层甚至 2 层，不仅是少写了代码，更重要的是减少了内存拷贝（Memory Copy）和对象创建的开销，这对于高性能场景是隐形的性能提升。
2. **关于放弃 `webrtc-rs`：** 这是一个非常成熟的架构师级别的决策。`webrtc-rs` 虽然是纯 Rust 实现，但在 Android 端涉及硬件编解码（MediaCodec/Surface）时，纯 Rust 库往往比较尴尬，需要大量的 FFI 甚至 unsafe 代码去操作 JNI。及时止损，寻找更适合 Android 生态的方案是明智的。
3. **关于 GPUI/wgpu：** 这与你之前的“Rust 主权架构”完美契合。如果未来你想摆脱 Android View/Compose 的限制，直接用 Rust 画 UI，这两个库是必经之路。
