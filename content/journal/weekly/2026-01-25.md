## 架构与 FFI：从“绑定”到“原子积木”

你对 **Rust FFI** 的反思非常深刻。过去我们容易把 FFI 接口写得太“业务化”，导致 UI 变了，底层也要跟着动。

- **集中式 vs 分布式 FFI：**
  - **集中式（旧）：** 一个巨大的 `LibApi` 接口，所有的业务逻辑都塞进去，维护成本极高。
  - **分布式（新）：** 将 Rust 层抽象为原子化的 **Service 积木**（如 `PostService`, `TopicService`）。
- **胶水层（Android ViewModel）：**
  - ViewModel 不再承载核心逻辑，只负责**组合（Compose）**这些 Rust 积木。
  - 利用 **UniFFI** 自动生成的绑定，保持 Rust 实体与 Kotlin 实体的一致性。

## 🕹 Selkies 与云游戏协议对接

对接过程中的“人为损耗”（后台不清楚双 WebSocket 连接）确实很打击积极性，但这也是技术成长的机会。

- **Selkies 双连接谜题：** 既然已经接通，下一步建议分析这两条 WS 的职责分工。
- **WebRTC 40s 断开问题：** * 这通常是 **ICE Keepalive** 或 **STUN/TURN 权限过期** 导致的。
  - 建议排查：是否因为第二条 WS 未能及时响应心跳，导致 Selkies 后端认为客户端已离线。

## 🧠 技术深度：TypeState 与 Actor 模型

你提到的 **TypeState (类型状态)** 是 Rust 的杀手锏，非常适合处理像“游戏接入排队”这种复杂的业务流。

- **TypeState 优势：** 在编译期拦截非法操作。比如：处于 `Queueing` 状态的玩家，绝对无法调用 `StartGame` 方法。
- **MVI + Actor：** * UI 发出 Intent -> Actor 接收并处理（Rust 层）-> 返回新的 State。
  - 这种模式能完美解决并发冲突，尤其是处理 WebRTC 多连接状态时。

## 🎨 UI 细节与工程化

- **GPUI / Wgpu：** 尝试用 Rust 直接触达 GPU 渲染，这与你长期的云游戏计划（客户端渲染生成鼠标等）高度契合。
- **ContentPadding vs Padding：** * 在 Compose 中，`contentPadding` 决定了子元素（如 LazyColumn 里的 Item）相对于边界的间距，而 `padding` 则是整个容器的缩进。
- **Flow.first()：** 这是一个关键的意识转变。在 `refreshNow` 等一次性操作中，坚决不使用持续订阅，以节省内存开销。
